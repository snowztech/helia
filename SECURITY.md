# Security policy

We take security seriously. If you find a vulnerability in Helia, please
report it to us privately so we can fix it before public disclosure.

## Reporting a vulnerability

Email **gethelia@protonmail.com** with:

- A description of the issue
- Steps to reproduce
- Affected versions if known
- Your assessment of the impact

We aim to acknowledge reports within **3 business days** and to ship a
fix or mitigation for confirmed high-severity issues within **30 days**.

If the issue is critical and you have not heard back within a week, feel
free to send a follow-up.

## Scope

In scope:

- Authentication and session handling (admin, API)
- Per-workspace data isolation
- HMAC identity verification flow
- Encryption-at-rest of secrets (identity secret, tool headers)
- Rate-limiting and quota enforcement
- The widget bundle (`@helia/widget`)
- SQL injection, XSS, CSRF, SSRF
- Dependency CVEs that affect the runtime

Out of scope:

- Issues that require physical access to a server you control
- Self-hosted misconfigurations (run with a proper reverse proxy, set
  the documented env vars, do not expose your database)
- Reports generated by automated scanners without a working PoC
- Social engineering or phishing of project maintainers

## Disclosure

Once a fix lands and a release is cut, we credit the reporter in the
release notes unless they prefer to remain anonymous.

## Self-hosters

Helia is distributed under AGPL-3.0. You run your own deployment, and
the responsibility for keeping it patched is yours. We publish security
fixes as tagged releases. Subscribe to GitHub releases to be notified.
