# V1

The plan to ship a Helia your first paying customer can self-host and use
in production. Today (2026-05-22), the product is feature-rich but not
deployable. Five sprints close that gap.

This file supersedes the older `MVP.md` and `PLAN.md`, which captured the
2026-05-21 framing before the polish work happened. They stay in the repo
as history.

## Vision for v1

> A company stands up Helia in 10 minutes (`docker compose up`), brands
> their bot, plugs in their APIs, and embeds the widget in their app where
> their users — verified by identity — chat with an AI assistant that has
> access to the company's own data.

## Status today

What's shipped, working, typechecked, themed:

- **Sources** — PDF, URL crawl, text upload → chunked, embedded, hybrid retrieval (vector + FTS + RRF)
- **Tools** — HTTP endpoints registered via admin, agent calls them dynamically. Built-in `search_knowledge`.
- **Widget** — Vanilla TS bundle (~20 KB), shadow-DOM isolated, themed via CSS vars. Brand from workspace config.
- **Admin UI** — Next.js + shadcn/ui + hugeicons. Pages: dashboard, sources, tools, widget, settings. Sticky headers, skeleton loaders, toast feedback, dark/light theme.
- **Observability** — `chat_traces` table, `/v1/metrics`, `/v1/conversations`. Recent messages list with source-count badges.
- **Settings** — workspace name, locale, model selector, system info, allowed origins (read-only).
- **Codebase** — TS strict everywhere, no `any` floating, monolith per the architecture decision, 8 routes building clean.

What's missing for production use is grouped into the five sprints below.

## The five sprints

Each sprint is independently shippable: one PR, one feature flag if needed,
no half-merged state. Definition of done at the end of each.

### Sprint 1 — Deploy bundle  ·  ~4 hours

Goal: a customer can `docker compose up`, set their OpenAI key, and have a
working Helia at `localhost:3000` and `localhost:4000` within 2 minutes.

- `apps/api/Dockerfile` — multi-stage, pnpm workspaces, `tsx` runtime
- `apps/web/Dockerfile` — multi-stage, Next.js standalone output
- `docker-compose.yml` — adds api + web services to existing postgres
- Schema auto-apply on api boot (waits for postgres, `drizzle-kit push`, runs `db:init`)
- Widget bundled into `apps/web/public/w.js` via prebuild script
- `apps/web/next.config.ts` — `output: 'standalone'`, `outputFileTracingRoot` for monorepo
- `apps/api/.env.example` — production hints (`DATABASE_URL`, `HELIA_CORS_ORIGIN`)
- `SELF_HOST.md` — step-by-step deploy guide
- `make docker-up` / `make docker-build` / `make docker-down` targets
- `.dockerignore`

**Definition of done**

- `docker compose up` from a fresh clone, with only `OPENAI_API_KEY` set, starts all three services
- Admin reachable at `localhost:3000`, API at `localhost:4000`
- Widget bundle served at `localhost:3000/w.js`
- Schema is applied automatically on first boot; default workspace exists
- `SELF_HOST.md` walks a non-author through the setup with no missing steps

### Sprint 2 — Admin auth  ·  ~2 hours

Goal: only the workspace owner can edit the admin. No one else with the URL
can fiddle.

- `better-auth` integration with email magic link
- One owner per workspace (self-host MVP)
- All `/v1/*` admin routes (sources, tools, workspace, settings, conversations) gated by session
- `/v1/chat` and `/v1/widget/config` stay public (used by visitors)
- Sign-in page at `/login`
- `RESEND_API_KEY` or `SMTP_*` env vars documented (or console-print magic links when missing — dev mode)

**Definition of done**

- A fresh visitor on `/` is redirected to `/login`
- After magic-link sign-in, the workspace owner sees the admin
- Logging out clears the session
- Public endpoints (`/v1/chat`, `/v1/widget/config`) keep working unauthenticated
- Chat from the widget still works for end users

### Sprint 3 — Identified user mode  ·  ~4 hours

The killer feature for fitdistance and similar B2B SaaS. The widget passes
a signed user identity to Helia; tools scope to the verified user.

- `workspaces.identity_secret` — HMAC secret, encrypted at rest with `MASTER_KEY`
- `chat_traces.user_id`, `chat_traces.user_name` columns
- Middleware on `/v1/chat`: verifies `X-Helia-User` + `X-Helia-Signature` HMAC against the workspace secret
- Tool execution honors `paramsSchema[].source === 'context'` — injects from `ctx.user.id`, ignores LLM-supplied value
- Widget exposes `Helia.identify({ id, name, signature })` JS API
- Widget supports `data-token-endpoint="/api/helia-token"` — auto-fetches user info, retries on 401
- `/settings` section "User identity" — toggle enable, generate/copy/regenerate secret, code samples per stack (Node, Next.js, Express, Rails, Django, PHP), live test endpoint
- Dashboard groups recent messages by `user_id` (real name shown, not made-up initials)

**Definition of done**

- Customer follows the docs: enables identified mode, copies secret, adds a backend route, adds a script tag with `data-token-endpoint`
- Widget mounts, fetches the token, sends signed user_id on chat
- Helia verifies the HMAC, rejects tampered tokens with 401
- A tool with `coach_id` configured as `source: "context"` is called with the verified user_id, not whatever the LLM proposed
- Dashboard shows messages grouped per user

### Sprint 4 — Production safety  ·  ~3 hours

Stops runaway costs and protects customer secrets at rest.

- Rate limiting on `/v1/chat`:
  - Per IP: 30 req/min
  - Per workspace: configurable, default 1000 req/hour
  - Per identified user: configurable, default 100 req/hour
- Tool `headers` JSONB column encrypted at rest with `MASTER_KEY` (AES-256-GCM); admin UI masks values, shows "set ●●●●", offers rotate
- Token usage cap per workspace per month (default 2M tokens, 402 when exceeded)
- Admin alerts (banner on `/`) when approaching limits

**Definition of done**

- Spamming the chat endpoint from one IP returns 429 after the limit
- A workspace that hits its token quota returns 402 with a clear message
- Existing tools with plaintext headers are migrated to encrypted on first save (one-time)
- Settings page shows current usage and quotas

### Sprint 5 — Conversations / moderation  ·  ~4 hours

The customer can see what their users are saying and intervene.

- `/conversations` list page with filters: by user, by date range, has-error, low-score
- `/conversations/[id]` detail: full trace — user message, agent reply, retrieval scores per chunk, tool calls with args/results, latency, tokens, model
- Replay button: re-run the same question against current sources
- CSV export (one row per turn)
- "Flag for review" toggle on a conversation (no workflow yet, just a marker)

**Definition of done**

- Customer can find a specific bad answer in under 30 seconds
- Drill-in shows everything needed to debug: was retrieval bad? Did the tool fail? Did the model hallucinate?
- A flagged conversation is visible at the top with a marker
- Export CSV opens cleanly in Excel/Sheets

## Best practices applied

These rules govern every sprint.

### Code

- TypeScript strict everywhere, no `any` without a comment explaining why
- Tests on the critical paths added per sprint: chat stream parsing, tool dispatch, HMAC verify, rate limit boundary
- `pnpm typecheck` and `pnpm -r build` must pass before any merge
- Lint via `next lint` and ESLint flat config on every PR
- Conventional Commits: `type(scope): subject` — `feat`, `fix`, `refactor`, `docs`, `chore`. One-line subject, body in the PR description

### Database

- After sprint 1, no more `drizzle-kit push` in production. Sprint 1 generates the first migration; every schema change after is a migration file checked into the repo.
- Reversible migrations: every `up` has a `down`
- Run migrations as part of the boot sequence (the same `start.sh` that's added in sprint 1)

### Security

- No secrets in DB plaintext after sprint 4 (tool headers, future API keys)
- Single `MASTER_KEY` env var per deployment, used for AES-GCM. Rotation procedure documented.
- HMAC for identified users (sprint 3)
- Rate limiting (sprint 4)
- Admin auth (sprint 2)
- No CORS wildcard in production — `HELIA_CORS_ORIGIN` is enforced

### Deployment

- One `docker compose up` brings up the full stack
- All config via env vars (12-factor)
- Healthchecks on every service
- Logs to stdout (JSON via pino)
- Graceful shutdown on SIGINT/SIGTERM (already in place)

### Documentation

- Every sprint ships with the doc updates next to the code (no "I'll write docs later")
- `SELF_HOST.md` updated by sprint 1
- `/settings` UI snippets updated by sprint 3
- `README.md` API table updated whenever a route is added

### Observability

- Every request logged via pino with `traceId`
- Every chat persists a `chat_traces` row (already shipped)
- Errors surface to the admin via toast OR a banner on the dashboard
- Sentry integration is optional via `SENTRY_DSN` env (lands in sprint 4 or later)

### UX

- Sticky page header pattern applied where there's a primary action
- Skeleton loaders on every data fetch
- Real error messages with recovery hints, not "Unauthorized"
- shadcn/ui + hugeicons stays the consistent vocabulary

## What's NOT in v1 (intentionally)

These are real but they wait until customers ask:

- Conversation memory across sessions (sliding window, summarization, vector recall) — current full-history-per-call works for short conversations
- Sentiment scoring / thumbs up-down on widget — moderation page is enough
- Multi-workspace from one Helia install — self-host = single workspace MVP
- Mobile responsive admin — desktop-only is fine for owners
- Slack / WhatsApp channels — widget covers the use case
- Workflow builder
- Eval harness
- Multi-provider LLMs beyond OpenAI — env var lands when there's demand

## After v1

V1.5 candidates, decided by what the first paying customers ask for:

- `@helia/server` + `@helia/react` npm packages (DX polish — 1-line identify integration)
- Memory tuning (sliding window + summarization) — when conversations get long
- Multi-workspace per install
- Conversation grouping by session (column exists, widget needs sessionStorage)
- Mobile responsive
- Stripe billing (cloud only)

V2: driven by reality, not guesses.
